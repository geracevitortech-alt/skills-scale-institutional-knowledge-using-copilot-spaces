## Purpose
This checklist ensures consistent, proactive stakeholder communication throughout the project lifecycle. Use it to plan, execute, and track stakeholder engagement activities.

---

## Phase 1: Stakeholder Identification and Planning

### Stakeholder Mapping
- [ ] Identify all internal stakeholders (teams, departments, executives)
- [ ] Identify all external stakeholders (customers, partners, vendors)
- [ ] Document stakeholder interests and concerns
- [ ] Assess stakeholder influence and impact on project
- [ ] Classify stakeholders by engagement level needed (high/medium/low)
- [ ] Create stakeholder contact list with preferred communication channels

### Communication Planning
- [ ] Define communication objectives for each stakeholder group
- [ ] Determine communication frequency (daily/weekly/monthly/milestone-based)
- [ ] Select appropriate communication channels (email, meetings, dashboards, reports)
- [ ] Identify communication owners and backup contacts
- [ ] Create communication calendar with scheduled touchpoints
- [ ] Define escalation path for urgent communications
- [ ] Establish feedback collection mechanisms

---

## Phase 2: Project Initiation Communication

### Project Launch
- [ ] Send project kickoff announcement to all stakeholders
- [ ] Schedule and conduct stakeholder kickoff meeting
- [ ] Present project charter/one-pager with goals and success metrics
- [ ] Share project timeline and major milestones
- [ ] Clarify roles, responsibilities, and decision-making authority
- [ ] Set expectations for ongoing communication cadence
- [ ] Establish regular meeting schedules
- [ ] Share access to project documentation and status dashboard

### Initial Alignment
- [ ] Confirm stakeholder availability for key milestones
- [ ] Collect initial feedback and concerns
- [ ] Document stakeholder expectations and requirements
- [ ] Address questions about project scope and approach
- [ ] Validate communication plan with stakeholders

---

## Phase 3: Ongoing Project Communication

### Regular Updates (Weekly/Monthly)
- [ ] Prepare status update using standard template
  - [ ] Progress this week/period
  - [ ] Achievements and milestones reached
  - [ ] Next steps and upcoming milestones
  - [ ] Risks and blockers
  - [ ] Decisions needed
- [ ] Distribute status update via agreed channels
- [ ] Update project dashboard with current metrics
- [ ] Respond to stakeholder questions within 24 hours
- [ ] Track stakeholder engagement and feedback

### Milestone Communications
- [ ] Announce upcoming milestones 2 weeks in advance
- [ ] Send reminder 3 days before milestone
- [ ] Conduct milestone demo/review with stakeholders
- [ ] Share milestone achievements and learnings
- [ ] Collect feedback on milestone deliverables
- [ ] Document stakeholder sign-off or approval
- [ ] Communicate next milestone objectives

### Risk and Issue Communication
- [ ] Notify stakeholders of new high-priority risks within 24 hours
- [ ] Provide risk mitigation plans and status updates
- [ ] Escalate critical risks to executive stakeholders immediately
- [ ] Communicate blocker status and resolution efforts
- [ ] Share lessons learned from resolved issues
- [ ] Update stakeholders when risks are closed or mitigated

### Change Communication
- [ ] Notify stakeholders of proposed changes requiring input
- [ ] Share change impact analysis and recommendations
- [ ] Collect stakeholder feedback on change requests
- [ ] Communicate change control decisions and rationale
- [ ] Update project documentation with approved changes
- [ ] Explain implementation timeline and impact

---

## Phase 4: Critical Event Communication

### Incident/Crisis Communication
- [ ] Activate incident communication plan immediately
- [ ] Notify key stakeholders within [specified time]
- [ ] Provide situation summary and current status
- [ ] Share actions being taken to resolve issue
- [ ] Set expectations for next update
- [ ] Send regular updates every [specified interval]
- [ ] Announce resolution and post-incident summary
- [ ] Schedule blameless retrospective with stakeholders

### Schedule/Scope Changes
- [ ] Explain the reason for the change
- [ ] Detail impact on timeline, budget, or deliverables
- [ ] Present mitigation options and recommendations
- [ ] Obtain necessary approvals and sign-offs
- [ ] Communicate revised plan to all affected parties
- [ ] Update all project documentation and artifacts

---

## Phase 5: Release Communication

### Pre-Release Communication
- [ ] Announce release schedule 2 weeks in advance
- [ ] Share release notes draft for stakeholder review
- [ ] Conduct release readiness review with stakeholders
- [ ] Confirm deployment window and maintenance communications
- [ ] Prepare rollback communication plan (if needed)
- [ ] Notify customer-facing teams (support, sales, success)
- [ ] Brief executive stakeholders on release scope and risks

### Release Announcement
- [ ] Send release announcement to all stakeholders
- [ ] Highlight key features and improvements
- [ ] Provide links to release notes and documentation
- [ ] Share training materials or user guides (if applicable)
- [ ] Announce support channels for questions
- [ ] Set expectations for monitoring and follow-up

### Post-Release Communication
- [ ] Share deployment success confirmation within 24 hours
- [ ] Report on post-deployment metrics and smoke tests
- [ ] Communicate any issues identified and resolution status
- [ ] Collect stakeholder feedback on release
- [ ] Send thank-you message to project team and stakeholders

---

## Phase 6: Project Closure Communication

### Final Deliverables
- [ ] Present final project outcomes and metrics
- [ ] Share complete project documentation
- [ ] Deliver final status report
- [ ] Document lessons learned and recommendations
- [ ] Obtain formal stakeholder sign-off on deliverables

### Closure Activities
- [ ] Conduct project retrospective with stakeholders
- [ ] Share retrospective summary and action items
- [ ] Celebrate project success with team and stakeholders
- [ ] Transition ongoing support or maintenance ownership
- [ ] Archive project communications and documentation
- [ ] Send project closure announcement and thank-you notes

### Continuous Improvement
- [ ] Share key learnings with broader organization
- [ ] Update process documentation based on insights
- [ ] Propose improvements to stakeholder engagement approach
- [ ] Document stakeholder feedback for future projects

---

## Communication Quality Standards

### Every Communication Should Include:
- [ ] Clear subject line or purpose
- [ ] Audience-appropriate level of detail
- [ ] Specific action items or decisions needed (if applicable)
- [ ] Deadline or timeframe for response (if applicable)
- [ ] Contact information for questions
- [ ] Links to relevant documentation or dashboards

### Communication Best Practices:
- [ ] Use consistent formatting and templates
- [ ] Lead with the most important information
- [ ] Be concise and action-oriented
- [ ] Use visuals (charts, dashboards) when appropriate
- [ ] Proofread before sending
- [ ] Send through appropriate channels
- [ ] Follow up on action items
- [ ] Archive important communications
